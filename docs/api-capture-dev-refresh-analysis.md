# Next.js 开发环境下「自动刷新导致录制失效」问题分析

## 一、现象

在 `npm run dev` 开发环境中，使用「API 采集 / 录制」时页面会**自动刷新**，导致：

- 录制中的 SSE 连接断开
- 界面显示或行为异常，用户感觉「录制失效」

## 二、根本原因

### 1. Next.js 开发模式会触发页面更新

在 `next dev` 下：

- **文件监听**：项目内任意非 ignore 的文件被修改并保存后，会触发重新编译。
- **更新方式**：
  - **Fast Refresh（HMR）**：只替换部分模块，尽量保留 React 状态，**不整页刷新**。
  - **Full Reload（整页刷新）**：当无法做安全的 HMR 时，会整页重载。

以下情况容易触发 **整页刷新**（而不是局部 HMR）：

- 修改了 **根 layout**（`app/layout.tsx`）、`_app`、或顶层 Provider
- 修改了 **环境变量** 或 **next.config**
- 修改了 **服务端组件** 或 **服务端相关逻辑**
- 某些 **依赖/模块图** 变更导致 HMR 无法安全更新
- 使用 **Turbopack** 时，部分场景下会直接做 full reload

因此：**只要在开发时改代码并保存，就有概率导致当前打开的页面（包括 /api-capture）被整页刷新。**

### 2. 刷新如何导致「录制失效」

录制功能依赖两条线：

| 角色 | 作用 |
|------|------|
| **前端页面**（/api-capture） | 展示录制状态、已抓到的请求列表；通过 **SSE** 接收服务端推送的新请求 |
| **后端录制器**（Playwright 等） | 真正负责抓包、生成 HAR、通过 SSE 推给前端 |

当 **前端页面** 发生整页刷新时：

1. 当前页面的 React 组件**整体卸载**。
2. `useRecordingSSE` 的 `useEffect` 执行 **cleanup**，会关闭当前的 **EventSource**（SSE 连接断开）。
3. 所有 **React 状态**（如 `isRecording`、`capturedApis`、`session`）被清空。
4. 新页面重新挂载后：
   - 会重新执行「恢复录制状态」的逻辑（如读 localStorage + 调 `/api/recording/status`）。
   - 若恢复成功，会重新建立 SSE；若恢复失败或超时，界面上就表现为「没在录制」或「列表不更新」。

所以：

- **后端录制器** 有可能还在跑（尤其是浏览器录制窗口没关），但 **前端已经拿不到新请求**，用户会认为「录制断了」。
- 若恢复逻辑不完善或网络/时序有问题，刷新后界面可能一直显示未录制或旧数据，即 **「录制失效」**。

### 3. 为何会「总是」自动刷新

- 开发时 **频繁保存文件**（改样式、改组件、改 API 等），每次保存都可能触发一次编译。
- 若改动经常涉及 layout、config 或服务端相关代码，就会 **经常触发整页刷新** 而不是 HMR。
- 若使用 Turbopack，当前版本在某些场景下更容易整页重载。

因此「总是自动刷新」= 开发模式下的 **编译 + 更新机制** 与 **当前修改的文件类型/范围** 共同作用的结果，而不是录制页独有的逻辑错误。

## 三、结论汇总

| 问题 | 原因 |
|------|------|
| 为什么会自动刷新？ | Next.js dev 监听文件变化并重新编译；部分变更会触发整页重载而非 HMR。 |
| 为什么一刷新就「录制失效」？ | 刷新会卸载页面、断开 SSE、清空 React 状态；若恢复失败或未及时重连，前端就收不到新请求。 |
| 为什么感觉「总是」发生？ | 开发时频繁保存代码，且可能常改到会触发 full reload 的文件。 |

根本原因可以概括为：**开发环境的「自动编译 + 整页刷新」与「依赖长连接（SSE）和前端状态的录制功能」不兼容**——刷新会主动断开连接并重置状态。

## 四、可行方案（建议）

1. **录制时尽量少改代码**  
   需要长时间录制时，尽量不保存会触发整页刷新的文件（如 layout、config、顶层组件）。

2. **强化「刷新后恢复」逻辑**（推荐）  
   - 刷新后根据 localStorage + `/api/recording/status` 自动恢复「正在录制」状态并**立即重连 SSE**。  
   - 当前项目已有部分恢复逻辑，可检查：超时时间、错误重试、以及「恢复中」的 UI 提示，确保用户刷新后能无感或最少点击即可继续看到新请求。

3. **录制用生产构建**  
   使用 `npm run build && npm run start` 在本地起一个生产服务，用该窗口打开 /api-capture 做录制。生产模式不会因改代码而重新编译，自然不会有 dev 下的自动刷新。

4. **配置/代码层面减少不必要的刷新**（可选）  
   - 在 `next.config.mjs` 的 `watchOptions.ignored` 中已忽略 `prisma/dev.db` 等，可确认没有误监听到会频繁变动的文件。  
   - 避免在 layout 或顶层做频繁变更；与录制无关的调试尽量放在不触发 full reload 的叶子组件里。

5. **将录制页单独开窗口**  
   用新窗口打开 /api-capture，主窗口继续开发。这样即使主窗口因 HMR 刷新，录制窗口在未刷新的前提下不会断开 SSE；但若录制页本身被刷新，仍需要依赖方案 2 的恢复逻辑。

---

**总结**：  
- **原因**：Next.js 开发环境在文件变更后会重新编译，并可能整页刷新；刷新会断开 SSE 并清空录制页状态，导致前端收不到新请求或显示未录制。  
- **方向**：要么减少刷新（少改触发 full reload 的文件 / 用生产模式录制），要么接受刷新并做好「刷新后自动恢复录制状态 + 重连 SSE」。
